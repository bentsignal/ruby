# Auth And Worktree Goals

This document preserves the intent behind the auth/worktree overhaul. Future agents should read this before changing auth, URL config, Convex deployment config, worktree scripts, or local dev server setup.

## The Goal

Ruby should work well with git worktrees without requiring a new Google OAuth configuration for every local checkout.

Google OAuth requires authorized origins and callback URLs. Worktrees make that painful because each worktree can run on a different local URL. The intended fix is to use Portless so local web URLs are stable and predictable:

- Main checkout: `https://www.ruby.localhost`
- Linked worktree: `https://<worktree-id>.www.ruby.localhost`

The main development Convex deployment is the single shared authority for authentication. Worktree Convex deployments are for isolated application data only.

## Canonical URLs

Production:

- Convex cloud URL: `https://api.ruby.travel`
- Convex site URL: `https://site.ruby.travel`
- Web URL: `https://www.ruby.travel`

Main development:

- Convex cloud URL: `https://api.dev.ruby.travel`
- Convex site URL: `https://site.dev.ruby.travel`
- Web URL: `https://www.ruby.localhost`

Development worktree:

- Convex cloud URL: the worktree-specific Convex deployment URL generated by the worktree setup
- Convex site URL for auth: `https://site.dev.ruby.travel`
- Web URL: `https://<worktree-id>.www.ruby.localhost`

The important split is:

- Data reads/writes in a worktree should use that worktree's own Convex cloud deployment.
- Auth should continue to use the main development auth deployment/site, especially `https://site.dev.ruby.travel`, so Google only needs one registered development callback/auth surface.

## Current Repo State

These files currently encode pieces of this architecture:

- `packages/app-config/src/urls.ts` centralizes URL construction.
- `packages/app-config/src/overrides.ts` is intentionally a mutable/generated override file. Do not remove it just because it looks trivial.
- `scripts/worktree-db.sh` creates a worktree Convex deployment, writes `VITE_WORKTREE_ID` to `.env`, rewrites `packages/app-config/src/overrides.ts` with the worktree Convex cloud URL and worktree id, and does a one-shot Convex push so the isolated deployment has the current schema/functions before `pnpm run dev`.
- `scripts/worktree-init.sh` copies env files from the main checkout and runs `pnpm install`.
- `apps/web/src/features/auth/lib/server.ts` configures TanStack Start auth proxy helpers with `urls.convex.cloud` and `urls.convex.site`.
- `apps/web/src/app/api.auth.$.ts` proxies `/api/auth/*` through the Better Auth/Convex handler.
- `apps/web/src/app/auth.callback.tsx` handles the cross-domain one-time token callback and copies auth cookies.
- `packages/convex/src/auth.ts`, `packages/convex/src/auth.config.ts`, and `packages/convex/src/http.ts` are the Convex-side Better Auth integration.

Portless is installed intentionally. Do not infer from old history, removed config, or a lack of `portless.json` that Portless is no longer wanted.

## Portless Requirements

Portless is part of the intended architecture.

The desired web host is `www.ruby.localhost`, with worktree prefixes becoming `<worktree-id>.www.ruby.localhost`. Portless supports named `.localhost` HTTPS URLs and git worktree subdomain prefixes. Its docs describe `portless run --name myapp ...` producing `https://<branch>.myapp.localhost` in linked worktrees, and HTTPS is enabled by default.

For this app, the Portless base name should preserve the host shape `www.ruby`, not merely `ruby` or `web.ruby`. A worktree URL must be compatible with `createUrls()` in `packages/app-config/src/urls.ts`.

Do not replace the stable URL model with arbitrary ports, `localhost:3000`, or per-agent env vars unless the user explicitly changes this architecture.

## Better Auth And Convex Requirements

The project uses Convex + Better Auth with TanStack Start. The Convex guide at `https://labs.convex.dev/better-auth/framework-guides/tanstack-start` says the TanStack Start server should proxy auth requests to Convex, use the Convex cloud/site URLs in `convexBetterAuthReactStart`, wrap the root with `ConvexBetterAuthProvider`, and use `expectAuth: true` for a seamless authenticated initial render.

Keep these concepts intact:

- The web route `/api/auth/$` should remain the Better Auth proxy route.
- `ConvexBetterAuthProvider` should continue to wrap the app with the Convex client and Better Auth client.
- Auth cookies and cross-domain one-time-token handling are intentional. Do not delete `copyAuthCookies`, `withAuthCookies`, or `/auth/callback` as "extra" code without replacing that flow.
- Convex functions should use Convex auth identity (`ctx.auth.getUserIdentity()` or the existing `authedQuery`/`authedMutation` helpers) for app data access.
- If `expectAuth: true` is used, sign-out behavior should account for the docs' warning that a reload/redirect is usually needed after signing out.

## The Worktree Data/Auth Split

The main development deployment owns auth:

- Google OAuth client credentials live on the main development auth deployment.
- `BETTER_AUTH_SECRET`, `SITE_URL`, and other auth-related Convex env values must be present on the auth-owning deployment.
- Google should only need the stable main development auth/web URLs, not every worktree URL.

Each worktree gets its own Convex deployment for data:

- The deployment should have the same schema and functions.
- It should not copy production or main-development user data.
- It can safely diverge for schema experiments.
- It needs to trust tokens issued by the main development auth deployment.

The current `packages/convex/src/auth.config.ts` reflects this idea by adding the shared development auth deployment's real JWT issuer as a trusted provider for non-main deployments. Be very careful before changing that logic.

That trust must use Convex's `customJwt` provider shape, not the OIDC provider shape. The values come from the shared auth metadata endpoint at `https://site.dev.ruby.travel/api/auth/convex/.well-known/openid-configuration`, and `scripts/worktree-db.sh` writes them as:

- `SHARED_AUTH_JWT_ISSUER`: the metadata endpoint's `issuer`
- `SHARED_AUTH_JWT_JWKS`: the metadata endpoint's `jwks_uri`

The configured provider should effectively be:

- `issuer`: `SHARED_AUTH_JWT_ISSUER`
- `applicationID`: `convex`
- `algorithm`: `RS256`
- `jwks`: `SHARED_AUTH_JWT_JWKS`

This matters because the Better Auth Convex plugin mints a custom JWT whose issuer is the auth deployment's raw Convex site URL, even when the browser talks to the custom domain `site.dev.ruby.travel`. If a worktree deployment trusts `site.dev.ruby.travel` as the issuer, or trusts it as `{ domain, applicationID }`, Convex rejects the Better Auth token with `NoAuthProvider`.

## Source Of Truth

Use `packages/app-config/src/urls.ts` as the source of truth for app URLs.

Use `packages/app-config/src/overrides.ts` as the worktree-specific override hook. It exists so generated setup scripts can inject:

- `convexCloudUrl`: the worktree deployment's Convex cloud URL for app data
- `worktreeId`: the Portless/local URL prefix

The checked-in default should stay harmless:

```ts
export const convexCloudUrl: string | undefined = undefined;
export const worktreeId: string | undefined = undefined;
```

Worktree setup may rewrite this file locally. Do not remove it for being generated. Do not replace it with a pile of environment variables unless the user explicitly asks.

## Things Agents Must Not Do

- Do not remove Portless support.
- Do not remove `packages/app-config/src/overrides.ts`.
- Do not collapse all Convex URLs into one value. Cloud and site URLs have different jobs.
- Do not make worktree auth depend on registering a new Google origin/callback.
- Do not point worktree auth at the worktree Convex site unless the architecture has been intentionally changed.
- Do not configure shared development auth trust as an OIDC `{ domain, applicationID }` provider. Worktree data deployments must trust the shared development auth deployment as a `customJwt` provider using the issuer and JWKS discovered from the shared auth metadata endpoint.
- Do not remove the cross-domain Better Auth callback/cookie flow because it looks unusual.
- Do not replace stable local URLs with random ports.
- Do not assume missing committed setup means the intent was abandoned.

## Known Follow-Up Work

- Reinstall/configure Portless and commit the setup.
- Ensure the root/web dev scripts launch the web app through Portless with the `www.ruby` base name.
- Confirm Portless's worktree prefix exactly matches the `worktreeId` generated by `scripts/worktree-db.sh`, or update one side so they agree.
- Confirm the Google OAuth authorized origins/callbacks include the stable main development URL(s) and production URL(s), not one-off worktree URLs.
- Recheck whether worktree Convex deployments need copied auth env vars or only trusted-provider config. Copying every env var from main development may be convenient, but it should not blur which deployment owns auth.
- Add/update tests or smoke checks for `createUrls()` so main development, worktree development, and production URL generation cannot regress silently.
